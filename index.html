Case Study Overview

```
📋 The Scenario CyberHealth Security has been hired to conduct a full penetration test on MedTechPro Hospital (MTPH) — a healthcare institution that relies on Electronic Health Records (EHRs), internal communications networks, and IoT-connected medical devices. The team must uncover vulnerabilities while maintaining strict ethical and operational standards.
🏥

Why MTPH is a High-Value Target

Hospitals hold extremely sensitive Electronic Health Records (EHRs) containing personal, medical, and insurance data. They also rely on IoT medical devices and 24/7 uptime, making disruption catastrophic and ransom payments likely.

EXAM FAVOURITE
🔐

What is a Penetration Test?

A simulated cyberattack authorised by the organisation to proactively identify exploitable vulnerabilities before real malicious actors find them. All actions require written consent via a Rules of Engagement (RoE) document.

DEFINE THIS
⚖️

The Ethical Challenge

CyberHealth must test live hospital systems without endangering patients, disrupting care, or exposing private medical data. They must balance thoroughness with the duty of care required in a healthcare environment.

ETHICS FOCUS
📡

Technology at MTPH

The hospital uses EHR databases, internal LANs, IoT medical devices (e.g. infusion pumps, monitors), web-facing portals for staff and patients, and email/communication systems — all of which are in scope.

CONTEXT

02

The 7 Phases of PTES

```

The Penetration Testing Execution Standard (PTES) is the structured framework CyberHealth follows. Each phase has a specific purpose — understand all seven and be able to apply them to the hospital context.

PHASE 01 Pre-Engagement Interactions +

Before any testing begins, CyberHealth must formally agree terms with MTPH. This includes defining the scope, obtaining legal written authorisation, and establishing the Rules of Engagement (RoE).

  • Define scope: which systems, networks, and devices are in/out of scope
  • Obtain signed authorisation (without this, penetration testing is illegal hacking)
  • Agree on testing hours to avoid disrupting critical patient care
  • Establish emergency contacts and rollback procedures
PHASE 02 Intelligence Gathering (Reconnaissance) +

The team collects as much public and semi-public information about MTPH as possible. This is called OSINT — Open Source Intelligence — and uses only legally available sources.

  • OSINT tools: search engines, social media (LinkedIn staff profiles), job postings, WHOIS lookups, DNS records
  • Search engine dorking: advanced Google operators to find exposed login portals or sensitive files (e.g. site:mtph.org filetype:pdf)
  • Network scanning: tools like Nmap to identify live hosts, open ports, and services
  • OS detection: identifying operating systems to find known unpatched vulnerabilities
  • Social engineering reconnaissance: phoning staff pretending to be IT to gather info (vishing)
PHASE 03 Threat Modelling +

Using the intelligence gathered, the team identifies the most likely attackers and their motivations, then prioritises which systems to test most carefully.

  • Identify adversary types: cybercriminals seeking EHR data, ransomware groups, disgruntled insiders
  • Map high-risk assets: EHR databases, IoT medical devices, admin portals
  • Prioritise patient-critical systems (e.g. ICU monitoring) for extra caution
  • Create a risk matrix to guide testing sequence
PHASE 04 Vulnerability Analysis +

The team actively scans for and catalogues weaknesses in MTPH's systems. This combines automated scanning tools and manual expert analysis.

  • Automated tools: Nessus, OpenVAS — scan for known CVEs, misconfigurations, unpatched software
  • Manual techniques: code review, configuration review, checking IoT device firmware versions
  • Identify SQL injection points in EHR web portals
  • Find XSS vulnerabilities in staff-facing web applications
  • Identify buffer overflow risks in legacy software
  • Extra caution applied to life-critical systems — use manual techniques only
PHASE 05 Exploitation +

The team attempts to exploit confirmed vulnerabilities to demonstrate real-world impact. All actions are carefully controlled and documented.

  • SQL injection to extract sample (non-real) patient records from test databases
  • Attempting to gain unauthorized access via stolen/default credentials
  • Exploiting weak IoT device authentication
  • Social engineering staff into revealing passwords (phishing emails, vishing calls)
  • Key constraint: must stop before causing actual harm; goal is proof-of-concept, not real damage
PHASE 06 Post-Exploitation +

Once inside a system, the team assesses what a real attacker could do — how far they could move, what data they could access, and what persistent access they could establish.

  • Lateral movement: testing whether a breach in one system allows access to others (e.g. from HR portal to EHR database)
  • Privilege escalation: attempting to gain admin rights from a standard user account
  • Data exfiltration simulation: demonstrating which patient data could be stolen
  • All findings logged meticulously; no real data extracted or altered
PHASE 07 Reporting +

The final phase produces a detailed, professional report for MTPH management and their IT team. This is the primary deliverable of the entire engagement.

  • Executive summary: non-technical overview for hospital leadership
  • Technical findings: each vulnerability documented with severity rating (critical/high/medium/low)
  • Evidence: screenshots, logs, and proof-of-concept demonstrations
  • Remediation recommendations: specific, actionable steps to fix each issue
  • Confidentiality: report must be stored and transmitted securely

💉

SQL Injection (SQLi)

Malicious SQL code inserted into a web form input field to manipulate the underlying database query. At MTPH, this could expose full patient EHR databases. Mitigated by prepared statements and input validation.

HIGH RISK AT MTPH
📜

Cross-Site Scripting (XSS)

Malicious JavaScript injected into a web page that executes in another user's browser. A stored XSS attack on MTPH's staff portal could steal session tokens, allowing attackers to impersonate medical staff.

SESSION HIJACK RISK
💾

Buffer Overflow

Writing more data to a memory buffer than it can hold, overwriting adjacent memory. Can allow arbitrary code execution. Legacy hospital software (old EHR systems) is especially vulnerable.

LEGACY SYSTEMS
🎭

Social Engineering

Exploiting human psychology rather than technical vulnerabilities. Includes phishing (deceptive emails), vishing (voice calls pretending to be IT support), and pretexting (fabricating a believable scenario).

HUMAN FACTOR
🦠

Malware Types

Malicious software includes ransomware (encrypts data for ransom — extremely common in hospitals), spyware (silently collects data), trojans (disguised as legitimate software), and rootkits (hide at OS level).

DEFINE EACH TYPE
🔍

Port Scanning & Network Mapping

Systematically scanning a network's IP range and ports to discover active devices, running services, and potential entry points. Tools like Nmap reveal the hospital's full network topology and open services.

RECON TOOL
🌐

OSINT

Open Source Intelligence — gathering data from publicly available sources: LinkedIn for staff names/roles, hospital website for software clues, job ads revealing tech stack, and Google dorking for exposed files.

PHASE 02
🖥️

Penetration Testing Types

Black box: tester has no prior knowledge (simulates external hacker). White box: full system info provided (most thorough). Grey box: partial info — balances realism with efficiency. MTPH uses grey box.

KNOW ALL THREE

05

Ethics & Professional Standards

```
⚠️ Exam Warning The 12-mark discussion question almost always centres on ethics. You must discuss BOTH sides — why ethical constraints can limit testing effectiveness, AND why they are non-negotiable in a healthcare context.

Authorisation & Consent

Every action must fall within the agreed scope. Testing systems outside scope — even if vulnerabilities are found — is illegal, regardless of intent. Written RoE must be signed before any testing.

Patient Data Confidentiality

CyberHealth must never exfiltrate, copy, or retain real patient data. If test systems must be used, they should contain anonymised synthetic data. GDPR and HIPAA-equivalent regulations apply.

Non-Disruption of Care

Testing must never interrupt life-critical systems. Aggressive automated scanning should not be used on ICU monitors or infusion pumps. Manual testing only for highest-risk patient-facing devices.

Responsible Disclosure

All vulnerabilities discovered must be reported only to MTPH — never disclosed publicly or to third parties before fixes are applied. The report itself must be transmitted and stored securely.

Minimising Footprint

The team should only access data and systems necessary to demonstrate a vulnerability. Going further than needed to prove a point is unethical and potentially illegal, even within an authorised test.

Consequences of Unethical Practice

Violations can result in criminal prosecution, civil liability, reputational damage to CyberHealth, regulatory fines for MTPH, and most critically — direct harm to patients if systems are disrupted.

💡 TOK Connection (for strong essays) The case raises the question: does the intent to help justify using hacking techniques? The answer in IB CS is nuanced — the same technical knowledge is ethically neutral; it is authorization, purpose, and constraint that define whether an action is ethical or criminal.

08

Exam Strategy & Tips

```
🎯

Always anchor answers to the MTPH context

Don't give generic cybersecurity answers. Every response should reference the hospital, patient data, EHRs, IoT medical devices, or the CyberHealth team specifically. Examiners award marks for contextualised application.

📊

Know the mark allocation command terms

"Identify" = 1 point each, no explanation needed. "Define" = precise technical definition. "Explain" = mechanism + why/how. "Describe" = detailed steps. "Discuss/Evaluate" = both sides + conclusion (needed for 8–12 mark questions).

⚡

For the 12-mark discussion question — use a structure

Write at least 4 distinct developed points covering multiple dimensions: technical ethics, patient safety, legal implications, professional responsibility, and the trade-off between thoroughness and harm avoidance. End with a clear conclusion.

🔧

Know the black/white/grey box distinction cold

This appears every year. Black box = external hacker simulation, no prior knowledge. White box = full access to code/architecture. Grey box = partial information (what MTPH uses). Know the advantages and disadvantages of each in a healthcare context.

📖

Re-read the case study the night before

You receive a copy of the case study in the exam. Annotate your pre-exam copy thoroughly — highlight phases, underline key terms, add marginal notes about PTES phases. You are expected to cite specific lines in strong answers.

⏱️

Time management in Paper 3

Paper 3 is 1 hour. With typically 30 marks of questions, aim for roughly 2 minutes per mark. Spend proportionally more time on the 12-mark essay. Don't overwrite short "identify" questions — 1 clear point per mark is sufficient.
